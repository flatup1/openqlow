## 1. 全体像：JIN の4要望

| # | 要望（JIN原文ニュアンス） | 一言で |
|---|---|---|
| ② | 「修正」と入れれば投稿内容を編集できるように | 本文をLINEで直す |
| ③ | 「挿入」と入れれば画像/動画を指定フォルダから挿入できるように | メディアを添える |
| ④ | 毎朝、候補を1つに絞り、内容と画像を選んで「今日はこの内容でいいですか？」と尋ねて | 朝の1候補確認 |

> ①は完全自動なので最後。②③④が揃い、かつ「投稿成功を検証する目」が完成してから解禁する。

## 2. 用語・前提

- **下書き / 候補（record）**：投稿前の本文＋メタ情報。`recordId` 例 `FG-YYYYMMDD-NNN`。保存先 `state/`。
- **承認語彙（既存）**：`OK <id>` 承認、`修正 <id>: …`、`×/やめる <id>` 却下、`ok`（直近候補）。本書はこれを拡張する。
- **半自動投稿**：画面を開き本文をクリップボードに置き、最後はJINが画面で投稿（`docs/browser-post-adapters.md`）。現在の標準。
- **完全自動投稿**：投稿ボタンまで自動。現在は安全ガードで停止中（`OPENQLOW_FORCE_FULL_AUTO` 拒否）。
- **メディアフォルダ**：`OPENQLOW_MEDIA_DIR`（デフォルト `~/openqlow/media/`）。画像/動画の候補置き場。

### 触ると想定される Codex 領域（COORDINATION.md §1）
- `src/commands/`（コマンドルーティング：修正/挿入/画像 N の解釈）
- `src/line_bot/`（LINE 受け答え・メッセージ整形）
- `src/approval/`（承認・再承認フロー）
- `src/scheduler/`（朝フローの起点）
- `src/distribution/` は **Claude領域**。本文生成・テンプレに変更が要る場合は **JIN 経由で Claude に依頼**（勝手に触らない）。
- `src/safety/`（修正後の本文再チェック）
- `src/state/`（下書き・選択状態の保存）
- `scripts/adapters/`（投稿実行・①の検証）

## 3. 実装順（推奨・強く推奨）

```
② 修正  →  ③ 挿入  →  ④ 朝の1候補確認  →  ① 完全自動（保留・最後）
```

| 優先 | 機能 | 安全度 | 依存 | 状態 |
|---|---|---|---|---|
| 1 | ② 修正で本文編集 | 高（投稿前テキストのみ） | なし | 未実装 |
| 2 | ③ 挿入でメディア添付 | 中（映り込みリスク） | なし | 未実装 |
| 3 | ④ 朝の1候補確認 | 中（②③を部品に使う・投稿しない） | ②③ | 未実装 |
| 4 | ① 完全自動投稿 | 低（誤投稿/BAN/誤成功） | 投稿成功検証 | **保留** |

理由：④は②③を内部で呼ぶので②③の後が作りやすい。①は事故リスクが高く、検証機構が前提。

## 4. ② 「修正」コマンド：本文編集

### ゴール
LINE で `修正 <新しい内容>` を送ると、承認待ちの下書き本文を書き換え、再確認に戻す。

### 決定事項
- **対象は直近の承認待ち1件**（複数あっても最新を対象）。`修正 <id>: …` で id 明示も許可。

### フロー
```text
JIN: 修正 セールは6/10までに変更
Bot: 下書きを更新しました。
     新しい本文: ＜更新後＞
     これでいいですか？ → OK / さらに修正 / NO
```

### 要件
1. 修正後は **必ず再承認を求める**（修正＝確定・投稿にしない）。
2. 修正後の本文に **`src/safety/` の安全チェックを再適用**（NGワード等を素通りさせない）。
3. 元本文を履歴として残す（誤修正のロールバック用、簡易でよい＝Codex判断）。
4. safety で弾かれたら、その旨をJINに伝えて再修正を促す（黙って確定しない）。

### 安全境界
- ❌ 「修正」した瞬間に投稿・確定は禁止。必ず再確認を挟む。

## 5. ③ 「挿入」コマンド：指定フォルダから画像/動画を添付

### ゴール
LINE で `挿入` を送ると、`OPENQLOW_MEDIA_DIR` のメディアを一覧 → 番号選択で下書きに添付する。

### 決定事項
- **フォルダ**：`OPENQLOW_MEDIA_DIR`、デフォルト `~/openqlow/media/`。
- **選択UI**：`挿入` で一覧表示 → `挿入 2` で2番目を選ぶ（番号方式）。

### フロー
```text
JIN: 挿入
Bot: ~/openqlow/media/ の候補
     1. kids_practice.jpg
     2. gym_front.jpg
     3. demo.mp4
     → 番号を送ってください（例: 挿入 2）
JIN: 挿入 2
Bot: 添付しました。
     本文＋添付（gym_front.jpg）を表示 → 目視確認して OK / 修正 / NO
```

### 要件
1. 対応拡張子はホワイトリスト（例 `jpg/jpeg/png/heic/mp4/mov`）。それ以外は弾く。
2. 添付後は **本文＋メディアをJINに表示して目視確認**を求める。
3. 添付情報は既存の `mediaFiles` 受け渡し（`scripts/adapters/lib/site_adapter.mjs` の `job.mediaFiles`）と整合させる。
4. フォルダが空・存在しない時は分かるメッセージを返す（無言で失敗しない）。

### 安全境界（FLATUP ルール・妥協不可）
- ⚠️ **挿入したメディアは必ず JIN が目視確認**してから投稿。
  お客様・お子さん・女性・会員さんの意図しない映り込みを防ぐため。
- ❌ フォルダ内のメディアを「無確認で自動添付・自動投稿」は禁止。
- 秘密情報：フォルダ走査で **パス以外の情報（EXIF 等）を不必要にログ出力しない**。

## 6. ④ 朝の「1候補だけ確認」フロー

### 背景
現状は朝に **3案/複数下書き**を出す（README "Daily three ideas"）。
JIN は「毎朝、本文1案に絞り、画像は候補から選び、1回だけ確認したい」。

### ゴール
毎朝、Bot が **本文を最有力1案に絞り**、投稿先と **画像候補（複数）** を添えて、
**「今日はこの内容でいいですか？」** と1メッセージで尋ねる。
**本文はBotが選ぶ。画像はJINが候補から番号で選ぶ。**

### 決定事項
- **本文は1案に絞って提示**（内部で複数案を作ってよいが、JINに出すのは1つ）。
- **画像は最初から複数候補（例 3〜5枚）を見せて、JINが番号で選ぶ**。自動で1枚に確定しない。
- **「画像なし」も選べる**。

### フロー
```text
Bot（朝）:
  今日はこの内容でいいですか？

  本文:
  ＜選んだ投稿文＞

  投稿先: all   ← all / threads / line_voom / google_business

  画像の候補（~/openqlow/media/ から）:
  1. kids_practice.jpg
  2. gym_front.jpg
  3. member_smile.jpg
  → 使う画像の番号を送ってください（例: 画像 1）。画像なしなら「画像なし」。

  OK   → この本文で進む（選んだ画像つき）
  修正 ＜文＞ → 本文を直す（§4へ）
  挿入 → 一覧にない画像を選ぶ（§5へ）
  NO   → 今日は出さない
```

### 要件
1. **本文1案に絞る**（最有力を選ぶロジックは Codex判断）。
2. **画像候補を複数提示**：`OPENQLOW_MEDIA_DIR` から数枚を一覧。多い時は新しい順などで絞ってよい（Codex判断）が、**自動確定はしない**。
3. **1メッセージにまとめる**：本文＋画像候補＋投稿先を「今日はこの内容でいいですか？」の中に収める。
4. 画像プレビュー手段（LINE image message でサムネ送付 等）は Codex判断。
5. **JINの応答語彙**（既存＋追加）：
   - `画像 N` → N番目を選択 → 選択後に確定版「この内容でいいですか？」を提示
   - `画像なし` → 画像なしで進む
   - `OK` → 次段へ（①未解禁の間は**半自動の投稿準備まで**。勝手に外部投稿しない）
   - `修正 …` → §4のフローへ（修正後はまた確認に戻る）
   - `挿入` → §5のフローへ（一覧外の画像。選び直したら確認に戻る）
   - `NO`（`×`/`やめる`）→ 今日は取り下げ
6. **候補がない朝**は無理に1件でっち上げない。「今日は候補なし」も正しい結果。

### 安全境界
- ❌ 朝の確認なしに自動投稿しない。④は「**1つに絞って尋ねる**」まで。
- ❌ 1候補に絞った結果を、JINが見ないうちに投稿確定にしない。**必ず `OK` を待つ**。
- ⚠️ 画像は必ず **最終的にJINが目視確認**（§5の映り込みルールを継承）。

### ②③①との関係
- ④は §4（修正）と §5（挿入）を **部品として呼ぶ** 画面。②③完成後が作りやすい。
- ①が将来解禁されても、④の「今日はこの内容でいいですか？」と最終 `OK` は **残す**（朝フローの本質）。

## 7. ① 「OK」で完全自動投稿（最後・保留）

### ゴール
LINE で `OK` を送ると、JINが画面で投稿ボタンを押さなくても投稿まで完了する。

### 決定事項
- **解禁時期**：②③④の完成後、かつ **投稿成功検証の完成後**に段階解禁。
  それまで `OPENQLOW_FORCE_FULL_AUTO` の拒否ガードは **外さない**。

### 前提条件（揃うまで有効化しない）
`docs/HANDOFF_20260607_claude→codex.md` §3-1 / §8 の **投稿成功検証**が必須。
- 投稿成立を **UI状態で確認できない場合 `status="posted"` を返さない**。
- 複数シグナル（URL変化 / DOM変化 / API確認 のうち最低2つ）で多重検証。
- 検証できなければ `status="uncertain"` か fail closed。

### 段階的な解禁手順
```text
1. Google Business に投稿成功検証を実装 → posted/uncertain/failed をテストで再現
2. LINE VOOM に投稿成功検証を実装 → 同上
3. 両方が安定して初めて OPENQLOW_FORCE_FULL_AUTO=true を解禁
4. 解禁初期は「自動投稿したが結果を必ずJINに報告」する運用から
```

### 安全境界（HARD・後退禁止）
- ❌ 「クリックできたから成功」禁止。
- ❌ 「タイムアウトしたから成功」禁止。
- ✅ 「投稿できていないのに成功扱いしない」を最後まで守る。
- 現状の `OPENQLOW_FORCE_FULL_AUTO=true` 拒否ガードは、検証完成まで維持。

## 8. 共通ルール（全機能・妥協不可）

1. **投稿できていないのに成功扱いしない**（最重要）。
2. 鍵 / トークン / Cookie / パスワード / セッションを **ログ・stdout に出さない**。
3. スクショは取らない（認証情報の映り込み防止）。
4. 公開操作（外部投稿・顧客LINE送信）は **必ずJIN承認後**。`OK` まで何も外に出ない。
5. push は **JIN 承認後**。コミットは `codex:` プレフィックス。
6. 新規ディレクトリ追加時は `COORDINATION.md §1` に **先に1行追記**してから実装。
7. `src/distribution/`（本文生成）に手を入れたくなったら **JIN経由でClaudeに依頼**（領域越境しない）。

## 9. 受け入れ条件（Definition of Done）

### 共通
- `npm run test` フルパス。
- 既存の半自動投稿・承認フローを壊していない。

### 機能別テスト
- **②**：修正 → 再承認が走る／修正後に safety 再適用される／safetyで弾かれたら再修正を促す。
- **③**：許可拡張子のみ添付／不正拡張子を弾く／空フォルダで分かるメッセージ／目視確認フロー。
- **④**：朝の本文が1案だけ／画像候補が複数並ぶ／`画像 N` で選べる／`画像なし` も選べる／本文＋画像候補＋投稿先が1メッセージに揃う／`OK`・`修正`・`挿入`・`NO` の分岐／候補なしの朝も成立。
- **①**（検証実装時）：`posted` / `uncertain` / `failed` の3分岐／検証不能時に `posted` を返さない。

### 完了条件
- ②③④が LINE 上で一通り動き、JIN が朝フローで「本文確認 → 画像番号選択 → OK」を体験できる。
- ①は本書 §7 の段階手順に沿って、検証完成まで **保留のまま**であること。
