## ② 「修正」コマンド：投稿内容の編集

### ゴール
LINE で `修正 <新しい内容>` のように送ると、承認待ちの下書き本文を書き換えられる。

### 期待フロー
```text
JIN: 修正 セールは6/10までに変更
Bot: 下書きを更新しました。新しい本文を表示 → 再度 OK/修正/NO を待つ
```

### 要件
- 対象は「いま承認待ちの下書き」1件（複数ある場合は recordId 指定 or 直近を対象、Codex判断）
- 修正後は **必ず再承認を求める**（修正＝自動で確定・投稿にしない）
- 元本文は履歴として残す（誤修正のロールバック用、Codex判断で簡易でよい）
- safety チェック（既存の `src/safety/`）を修正後の本文に **再適用**する

### 安全境界
- ❌ 「修正」で本文を変えた瞬間に投稿、は禁止。必ず再確認を挟む。

## ③ 「挿入」コマンド：デスクトップ指定フォルダから画像/動画添付

### ゴール
LINE で `挿入` と送ると、PC の決まったフォルダにある画像/動画を下書きに添付できる。

### 期待フロー（案、Codex判断で調整可）
```text
JIN: 挿入
Bot: media フォルダの候補を一覧表示（ファイル名 or 番号）
JIN: 挿入 2     ← 2番目を選ぶ
Bot: 添付しました。本文＋添付を表示 → JINが目視確認 → OK/修正/NO
```

### 要件
- 指定フォルダのパスは **環境変数で設定**（例：`OPENQLOW_MEDIA_DIR`、デフォルト `~/openqlow/media/`）
- 対応拡張子はホワイトリスト（例：jpg/png/heic/mp4/mov）。それ以外は弾く
- 添付後の本文＋メディアは **必ず JIN に表示して目視確認**を求める
- 既存の `mediaFiles` 受け渡し（`scripts/adapters/lib/site_adapter.mjs` の job.mediaFiles）と整合させる

### 安全境界（FLATUP ルール・妥協不可）
- ⚠️ **挿入したメディアは必ず JIN が目視確認**してから投稿。
  お客様・お子さん・女性・会員さんの意図しない映り込みを防ぐため。
- ❌ フォルダ内の画像を「無確認で自動添付・自動投稿」は禁止。
- 秘密情報：フォルダ走査でパス以外の情報（EXIF 等）を不必要にログ出力しない。

## ① 「OK」で完全自動投稿（最後・保留）

### ゴール
LINE で `OK` を送ると、JIN が画面で投稿ボタンを押さなくても投稿まで完了する。

### 前提条件（これが揃うまで有効化しない）
現在は半自動（`docs/browser-post-adapters.md` 参照）。完全自動の解禁には、
`docs/HANDOFF_20260607_claude→codex.md` §3-1 / §8 の **投稿成功検証**が必須。

- 投稿が成立したと **UI 状態で確認できない場合、`status="posted"` を返さない**
- 複数シグナル（URL変化 / DOM変化 / API確認 のうち最低2つ）で多重検証
- 検証できなければ `status="uncertain"` か fail closed

### 段階的な解禁手順（推奨）
```text
1. Google Business に投稿成功検証を実装 → テストで posted/uncertain/failed を再現
2. LINE VOOM に投稿成功検証を実装 → 同上
3. 両方の検証が安定したら、初めて OPENQLOW_FORCE_FULL_AUTO=true を解禁
4. それでも最初は「自動投稿したが結果をJINに必ず報告」する運用から
```

### 安全境界（HARD・後退禁止）
- ❌ 「クリックできたから成功」禁止
- ❌ 「タイムアウトしたから成功」禁止
- ✅ 「投稿できていないのに成功扱いしない」を最後まで守る
- 現状の `OPENQLOW_FORCE_FULL_AUTO=true` 拒否ガードは、検証完成まで外さない

## ④ 朝の「1候補だけ確認」フロー

### 背景（現状）
今は朝に **3つのアイデア / 複数下書き**を出している（README §"Daily three ideas"）。
JIN は「毎朝、候補を1つに絞って、投稿文＋画像をセットした状態で1回だけ確認したい」。

### ゴール
毎朝の下書き提示で、Bot 側が **最有力の1候補に絞り**、投稿内容と投稿画像まで選んだ上で、
**「今日はこの内容でいいですか？」**と JIN に尋ねる。JIN は1タップ感覚で答える。

### 期待フロー（案、Codex判断で調整可）
```text
Bot（朝）:
  今日はこの内容でいいですか？

  本文:
  ＜選んだ投稿文＞

  画像: ＜選んだ画像のファイル名＞（~/openqlow/media/ から自動選択）
  投稿先: ＜all / threads など＞

  OK → この内容で進む
  修正 ＜文＞ → 本文を直す（②へ）
  挿入 → 画像を選び直す（③へ）
  NO → 今日は出さない
```

### 要件
- **1候補に絞る**：複数案を内部で作ってよいが、JIN に出すのは1つだけ（最有力を選ぶロジックは Codex 判断）
- **投稿文を選ぶ**：その1候補の本文を確定状態で提示
- **投稿画像を選ぶ**：`~/openqlow/media/` から候補画像を1枚自動で添えて提示（③の仕組みを再利用）
- **1つの質問にまとめる**：「今日はこの内容でいいですか？」の1メッセージで本文＋画像＋投稿先を提示
- JIN の応答は既存の承認語彙に合わせる：`OK` / `修正 …` / `挿入` / `NO`(`×`/`やめる`)
  - `OK` → 次段（下書き保存 or 投稿準備。①が未解禁の間は**半自動の投稿準備まで**）
  - `修正 …` → ②のフローへ（修正後はまた「これでいいですか？」に戻る）
  - `挿入` → ③のフローへ（画像を選び直したらまた確認に戻る）
  - `NO` → 今日は提示を取り下げる

### 安全境界
- ❌ 朝の確認なしに自動投稿しない。④はあくまで「**1つに絞って尋ねる**」まで。
- ❌ 1候補に絞った結果を「JIN が見ていないのに投稿確定」にしない。必ず JIN の `OK` を待つ。
- ⚠️ 画像を自動で添える場合も、**最終的にJINが目視確認**（③の映り込みルールを継承）。
- 「今日は候補なし」の朝もありうる。無理に1件でっち上げない（薄い日は「今日は無し」も可）。

### ②③①との関係
- ④は ②（修正）と ③（挿入）を**部品として呼ぶ**画面。先に②③を作ると④が組みやすい。
- ①（完全自動）が将来解禁されても、④の「今日はこの内容でいいですか？」は残す。
  JIN の最終 `OK` は朝フローの本質なので削らない。
