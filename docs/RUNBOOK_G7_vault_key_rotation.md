# G7 手順書: Vault過去履歴の旧Google APIキー候補を失効・除去

> 対象リポジトリ: `flatup1/flatup`（Obsidian Vault = 記憶）
> 担当: 人間（鍵の失効）→ Codex（履歴書き換え）
> 重要: **秘密値・対象ファイル名をIssueやチャット・コミットへ書かないこと。**

中学生にもわかるように一言で言うと:
**「昔うっかりノート（Vault）に書いた“家の鍵の番号”が、消したつもりでも過去のページに残っている。だから①まず鍵そのものを作り直して古い番号を無効にし、②そのあとノートの過去ページからも番号を消す」** という作業です。①を先にやれば、②が終わるまでの間に番号が見られても安全です。

## なぜ必要か（採点 -7点の理由）
- 現在の最新（HEAD）には候補は **0件**。問題は **過去のコミット履歴**に残っている点。
- Gitは履歴を保持するので、`git log` や古いcloneから過去の値を取り出せてしまう。
- 影響: 旧Google APIキー候補 **2個**、関係ファイル **5個**（履歴上）。

## 前提（不可侵ルール）
- 失効が終わるまで履歴書き換えを急がない。**順序を守る**（失効が先、履歴浄化が後）。
- 秘密値そのものを画面共有・Issue・PR・チャットに貼らない。
- 自動化しない。各ステップは人間が確認してから次へ進む。

---

## ステップ1（人間のみ）: 鍵を無効化する ← 最優先・これだけで実害を止められる
1. Google Cloud Console → 「APIとサービス」→「認証情報」を開く。
2. 該当の旧APIキー2個を特定する（作成日が古いもの／用途不明のもの）。
3. 各キーで **「キーを削除」または「キーを再生成（ローテーション）」** を実行する。
   - まだ使われている可能性があるなら、先に新キーを発行して稼働中サービスへ差し替え → 旧キーを削除、の順。
4. 削除後、Google Cloud の「指標」で旧キーへのアクセスが0になることを数日観察する。
5. **結果（成功/失敗）だけ**をオーナーへ口頭またはプライベートに報告。値は報告しない。

> この時点で、たとえ履歴に値が残っていても **鍵は無効なので悪用できない**。採点上の実害（情報漏えいリスク）はここで大きく下がる。

## ステップ2（オーナー承認）: 履歴書き換えの許可
- ステップ1完了をオーナーが確認し、`flatup1/flatup` の履歴書き換えを**明示承認**する。
- 書き換えは全clone・全ブランチに影響するため、作業時間帯を周知する。

## ステップ3（Codex）: 履歴から値を除去
1. バックアップ用に現在のリポジトリをミラークローンしておく（ローカル・非公開）。
2. `git filter-repo`（推奨）で該当の文字列パターンを履歴全体から除去または置換する。
   - 例: `git filter-repo --replace-text <置換ルールファイル>`
   - 置換ルールファイルには値そのものを書く必要があるため、**そのファイルは作業端末ローカルのみ・終了後に削除**。リポジトリにコミットしない。
3. 書き換え後、`git log -p` 等で該当パターンが0件になったことを確認。

## ステップ4: 全cloneを再同期
1. 書き換え済み履歴を `--force-with-lease` でpush（オーナー承認済みの調整時間帯に実施）。
2. 既存の全clone（VPS・各自のPC・Codex環境）を破棄して再clone、または `git fetch && git reset --hard origin/<branch>`。
3. 古い参照（reflog, タグ）が残っていないか確認し、必要なら整理。

## ステップ5: 再スキャンで0件を確認（検証）
1. 再clone後のリポジトリで秘密スキャンを実行（openQLOWの `secret_guard` 相当、または `gitleaks`）。
2. **過去履歴を含めて** スキャンし、候補が0件であることを確認。
3. 結果（件数のみ）を `docs/STATUS_AND_GAPS.md` のG7に反映し、`status: done` に更新。

---

## 完了の定義（Doneチェックリスト）
- [ ] 旧キー2個が Google Cloud で削除/ローテーション済み（アクセス0を確認）
- [ ] オーナーが履歴書き換えを承認
- [ ] `flatup1/flatup` の全履歴スキャンで候補0件
- [ ] 全cloneが新履歴へ再同期
- [ ] `STATUS_AND_GAPS.md` のG7を `done` に更新（採点 +7 → セキュリティ30/30）

## やってはいけないこと
- 値そのものをIssue/PR/チャット/コミットメッセージに書く。
- 失効前に履歴だけ書き換える（無効化されていない鍵が他のcloneに残る）。
- 承認なしの強制push。
