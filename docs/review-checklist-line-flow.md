# レビュー観点チェックリスト（②③④⑤① 実装レビュー用）

Codex の実装が上がったとき、Claude（レビュー役）が確認する項目。
JIN もこのリストで「安全に出来ているか」を確認できる。
正本は `docs/HANDOFF_20260607_claude→codex_v2.md`。差異があれば指示書が優先。

---

## A. 最重要・安全境界（1つでも×なら差し戻し）

- [ ] 投稿できていないのに `posted`/「投稿しました」を返していない
- [ ] `OK` まで外部に何も出ない（朝フローでも勝手に投稿しない）
- [ ] 鍵 / トークン / Cookie / パスワード / セッションをログ・stdoutに出していない
- [ ] ⑤の受信メディアのバイナリ・取得URL・LINEトークンをログに出していない
- [ ] スクショを取っていない
- [ ] `OPENQLOW_FORCE_FULL_AUTO` の拒否ガードが外れていない（①は保留のまま）
- [ ] 画像/動画は投稿前に必ず JIN の目視確認を通る（③⑤④すべて）

## B. 指示書との整合（機能別）

### ② 修正
- [ ] `修正 <内容>` で直近の承認待ち1件の本文が変わる（`修正 <id>:` 指定も可）
- [ ] 修正後に再承認を求める（即確定・即投稿しない）
- [ ] 修正後の本文に safety 再適用、弾かれたら再修正を促す
- [ ] 元本文が履歴として残る

### ③ 挿入（フォルダ）
- [ ] `OPENQLOW_MEDIA_DIR`（デフォルト `~/openqlow/media/`）を見る
- [ ] `挿入` で一覧 → `挿入 N` で選択（番号方式）
- [ ] 拡張子ホワイトリストで非対応を弾く
- [ ] 空フォルダ・不在で分かるメッセージ
- [ ] `job.mediaFiles` 経路と整合

### ⑤ LINE直送メディア
- [ ] image/video 受信 → content API でDL → 保存 → 添付
- [ ] 対応形式・サイズ上限で弾く、超過時メッセージ
- [ ] 正しい record に紐づく
- [ ] VPS→Mac の橋渡し経路が設計され、機密を漏らさない

### ④ 朝の1候補確認
- [ ] 本文は1案だけ提示（複数案は内部のみ）
- [ ] 画像候補を複数提示、`画像 N` で選択、`画像なし` 可
- [ ] LINE直送メディアでも差し替わる
- [ ] 本文＋画像候補＋投稿先が1メッセージ
- [ ] `OK`/`修正`/`挿入`/`NO` の分岐が動く
- [ ] 候補なしの朝が成立（でっち上げない）
- [ ] （任意）`docs/post-scoring-rubric.md` の採点軸で最有力を選んでいる

### ① 完全自動（実装した場合のみ）
- [ ] `posted`/`uncertain`/`failed` の3分岐
- [ ] 複数シグナル（最低2つ）で検証、検証不能で `posted` を返さない

## C. 領域・運用ルール

- [ ] `src/distribution/`（本文生成・Claude領域）に勝手に手を入れていない
- [ ] 新規ディレクトリ追加時に COORDINATION.md §1 へ追記
- [ ] コミットが `codex:` プレフィックス
- [ ] push は JIN 承認後（勝手にpushしていない）

## D. 品質

- [ ] `npm run test` フルパス
- [ ] 機能別テストが §9（受け入れ条件）を満たす
- [ ] 既存の半自動投稿・承認フローを壊していない

## E. FLATUPトーン（文面）

- [ ] 煽り/誇大/断定がない
- [ ] 「投稿できていないのに成功」と読める言い回しがない
- [ ] JIN が選べる言葉（OK/修正/挿入/画像N/NO）が添えられている
- [ ] `docs/line-copy-templates.md` と大きく矛盾しない

## 使い方
1. Codex の diff / ブランチ / ハンドオフ書を受け取る
2. A → B → C → D → E の順でチェック
3. A に×があれば即差し戻し（安全優先）
4. 結果を JIN に報告。曖昧な点は AskUserQuestion で確認
