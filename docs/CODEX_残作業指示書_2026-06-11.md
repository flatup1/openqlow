## タスク2【本丸】LINE webhook → CRM intake 配線

### 目的
PR #4で実装済みの接続口 `src/crm/line_intake.ts` の `intakeFromLineMessage()` を、**既存のLINE webhook受信処理に接続する**。新しいCRMは作らない。

### 接続口の実シグネチャ（これに合わせる）
```ts
import path from "node:path";
import { openProspectStore } from "../crm/store.js";
import { intakeFromLineMessage } from "../crm/line_intake.js";

const store = openProspectStore(
  path.join(process.env.OPENQLOW_DATA_DIR || path.join(process.cwd(), "data"), "prospects.json"),
);

// LINE の message(text) イベント1件につき:
const result = await intakeFromLineMessage(
  { lineUserId: event.source.userId, text: event.message.text /*, displayName 任意 */ },
  store,
  // now?: () => Date  受信日時を渡したい場合のみ
);
// result.prospect   … 登録/更新された見込み客
// result.replyDraft … オーナーが確認して送る返信案（★自動送信しない★）
// result.created    … 新規 true / 既存更新 false
```

### 実装方針
- LINE webhook で**テキストメッセージ受信時**に上記を1回呼ぶだけ。
- 渡す情報：`lineUserId`（必須・名寄せキー）/ `text`（必須）/ `displayName`（取れれば）。`contactSource` は接続口が "LINE" を自動付与。`now` で受信時刻を渡せる（省略可）。
- **重複防止は接続口側が `externalId`(=lineUserId) で実装済み**。webhook側で独自の重複管理を作らない。
- intake が例外を投げても webhook 全体を落とさない（try/catch）。必要なら `src/crm/self_repair.ts` の `logError("line_webhook_error", ...)` を呼ぶ。
- オーナー通知は**既存の push/approval の仕組みを再利用**して「新規問い合わせ #id ＋ 返信下書き」を提示（既存openQLOW承認フローと同じ思想）。

### 触ってよい範囲
- `src/line_bot/` の受信ハンドラ（メッセージ受信直後の分岐）
- 必要ならそのテスト追加

### 禁止事項
- **自動返信を実装しない／顧客へ勝手に送信しない／予約確定を自動化しない／料金変更しない**
- CRMを作り直さない・`src/crm/` を大きく変更しない
- SNS投稿生成を重複実装しない（既存 `generators/daily_three`・`mma_topic`→`distribution/expand`→`publish/*` が担当）
- Python / Flask / SQLite を新設しない

### 完了条件
- LINEで受信したテキストがCRMに登録される
- 同一LINEユーザーの再問い合わせは**重複登録ではなく更新**される
- **自動返信はされない**
- 既存のAIKA / openQLOW / CRM / SNSパイプラインを壊さない
- `npm run test` が通る
- 変更差分が最小・変更ファイル一覧とdiffを提示
