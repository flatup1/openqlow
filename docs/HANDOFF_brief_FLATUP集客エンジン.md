## 1. 背景・前提
- **FLATUP GYM**：千葉県成田市のキックボクシングジム。
- **ターゲット**：初心者女性／キッズ／運動不足の社会人／自信をつけたい男性／シニア。
- **強み**：女性インストラクター在籍／初心者に優しい／キッズ対応／初回体験500円／24時間サンドバッグ利用可／専用駐車場あり／大会「UIZIN」主催／ガチスパー禁止で安全重視／代表の熱量。
- **既存AI構成**
  - **AIKA**：顧客対応AI。LINE返信・問い合わせ対応・体験案内・FAQ。**自動送信は禁止**。人格は「FLATUP GYMの女性インストラクター・会長・代表補佐」。返信は基本「AIKA」で締める。
  - **openQLOW**：裏側の経営・集客AI（オーナー専用）。日次分析・返信漏れ確認・見込み客管理・改善提案・Obsidian保存。**自動送信禁止**。実体は**依存ゼロのTypeScript/ESM**（※Flask/Pythonではない）。

## 2. 最終ゴール
**FLATUP AI永久集客エンジン** ＝ 集客を半自動で回すAI。
**顧客を常に体験につなげるために、学習・記録・改善を続ける永久機関**を目指す。
「問い合わせ→体験→入会→口コミ→再来店」を、AIが**営業参謀**として前に進める（営業担当ではない／**最終判断は必ず人間**）。
AIが継続実行する：新規問い合わせ記録／見込み客分類／返信文生成／追客漏れ検知／体験予約候補抽出／体験後フォロー候補抽出／口コミ依頼候補抽出／入会率記録／失注理由分析／毎日の集客日報／エラー検知／自己修復ログ／改善案提示。

## 3. 絶対の禁止事項（最重要・厳守）
- AIが勝手に顧客へ送信しない
- AIが勝手に予約確定しない
- AIが勝手に料金変更しない
- AIが勝手に本番コードを書き換えない／大規模リファクタしない
- AIがしつこい営業文を作らない
- AIがLINEブロックにつながる追客をしない
- 既存のAIKA／openQLOW／LINE webhookを壊さない
- APIキーをコードに直書きしない／個人情報を不要に外部送信しない

## 4. 進め方の希望（オーナーの作業スタイル）
- **既存構成を壊さない・最小変更**で実装する。
- 現行システムと比べて**「差分のみ」追加**してブラッシュアップする。
- **コードを書き換える前に「どこをどう変えるか」必ず私に聞く**。
- **実機で動かして最適化**する。
- 説明は**「中学生にもわかるように」**。
- 重要判断は**「4人の専門家チーム（①リスク②マーケ③経営④現場コーチ）」で分析**してから動く。
- 人間確認前提／ログを残す／あとで拡張しやすくする。

## 5. 優先順位（今の核心）
> オーナー曰く：**「AIを賢くするより、記録する箱を作るのが最優先」**
1. **見込み客CRM**（無いと全部バラバラ。心臓部）
2. **追客候補の自動抽出**（返信忘れ防止）
3. **体験後フォロー**（入会率に直結）
4. **日報**（毎日やることが見える）
5. **自己修復ログ**（壊れた時に原因を残す）

## 7. やりたいツール群（FLATUP集客AI司令塔・全体像）
1. **問い合わせ返信AIKA**：問い合わせ文を貼ると、丁寧／短め／予約誘導強めの3返信＋24h後・3日後の追客文＋属性分類を生成。型は「共感→安心材料→FLATUPの強み→体験案内→候補日時」。
2. **体験予約コンバージョン管理**：名前/属性/目的/温度感/希望時間/次アクション/入会確度/最終連絡日を管理。24h返信なしを自動で拾う。
3. **体験後フォロー自動生成**：当日お礼／翌日フォロー／入会案内／口コミ依頼。体験メモから個別生成。
4. **Google口コミ依頼**：体験・入会・大会参加後の依頼文生成。
5. **SNS投稿ネタ自動生成**：Instagram/X/LINE/Googleビジネス向け。「映え」より「不安を消す」投稿。
6. **サイト改善AI**：公式サイトを読み、初心者女性が不安になる点／料金の分かりにくさ／体験予約ボタンの弱さ／キッズ保護者導線をチェック。
7. **広告文生成**：女性初心者／キッズ／40代男性／ダイエット／護身／運動不足 × Google/Instagram/LINE。

## 8. AIKA返信ルール
- 丁寧・やさしい／初心者の不安を消す／強引に売り込まない／最後に自然に体験へ誘導／絵文字は少なめ
- 料金を勝手に変更しない／日程を勝手に確定しない（**予約確定ではなく候補日の確認**にする）
- 女性・キッズ・初心者が安心する表現／必要に応じて最後を「AIKA」で締める

## 10. 見込み客CRM 仕様（MVP・最優先7項目）
> オーナー原文：「今回のMVPでは以下を最優先してください」
1. 見込み客を登録できる
2. ステータスを管理できる
3. 返信漏れを見える化できる
4. 体験後フォロー候補を出せる
5. 口コミ依頼候補を出せる
6. 毎日の集客日報を作れる
7. エラー時に自己修復ログを残せる
> 目的は完璧な自動化ではなく、**集客状態を毎日見える化し、AIKA/openQLOWが改善提案できる土台**を作ること。

**prospects（見込み客）カラム**：id / name / contact_source / gender / age_group / category / purpose / temperature / status / inquiry_text / ai_reply / human_sent_reply / last_contact_at / next_action / trial_date / trial_status / joined / lost_reason / memo / created_at / updated_at
**status**：new_inquiry / replied / waiting_reply / followup_needed / trial_scheduled / trial_done / joined / lost / archived
**temperature**：A（体験予約に近い）/ B（興味あり）/ C（情報収集中）
**category**：female / male / kids / parent_child / senior / unknown

**抽出条件**
- 追客漏れ（get_followup_needed）：status が waiting_reply か replied／最終連絡から24h以上／未入会／lost・archivedでない
- 体験後フォロー（get_trial_followup_needed）：status=trial_done／未入会／trial_date あり
- 口コミ依頼（get_review_request_candidates）：status=joined かつ joined=1

## 16. 実装時の最重要補足：Python/Flask化は禁止

このプロジェクトでは、以前の案に Flask / Python / SQLite Webアプリ案が含まれていたが、現在の正本方針は以下である。

* openQLOW は 依存ゼロの TypeScript / ESM で動いている
* Flask / Python の新規アプリを立てない
* 既存構成を壊さず、TSネイティブ最小CRM として差分追加する
* 既存のAIKA / openQLOW / LINE webhook の責務を混ぜない
* LINE受信側は別AI・別領域として扱い、openQLOW側は「受け取った問い合わせデータを記録・分類・日報化する」役割に集中する

今回の実装でやるべきことは、新しい大きなWebアプリを作ることではない。

やるべきことは、既存のopenQLOWに対して、以下を最小追加すること。

1. 見込み客CRMのデータ構造
2. 問い合わせ文から見込み客データを作る処理
3. ステータス管理
4. 返信漏れ・追客候補の抽出
5. 体験後フォロー候補の抽出
6. 口コミ依頼候補の抽出
7. 日次集客日報のMarkdown出力
8. 自己修復ログのMarkdown出力

禁止すること。

* Flaskを新設しない
* Pythonへ置き換えない
* SQLite前提で全面設計し直さない
* 既存ファイルを大規模リファクタしない
* AIKA / openQLOW / LINE webhook の責務を混ぜない
* 自動送信機能を実装しない
* 本番コードを勝手に修正しない

実装前に必ず以下を出すこと。

1. 触らない既存機能
2. 追加するファイル
3. 変更するファイル
4. データ保存先
5. 既存構成を壊さない理由
6. 実装後の確認手順

承認後に、差分のみで実装すること。
